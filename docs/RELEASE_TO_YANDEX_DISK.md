# Автодоставка релиза на Яндекс.Диск

Документ объясняет, как сделать так, чтобы **при каждой публикации релиза на GitHub**
установщик и портативный exe автоматически попадали в папку на Яндекс.Диске, а конечному
пользователю можно было дать **одну постоянную ссылку** на актуальную версию — без доступа
к приватному репозиторию.

Механизм: GitHub Actions (`.github/workflows/release-to-yandex-disk.yml`) ловит событие
публикации релиза, скачивает приложенные `*.exe` и заливает их на Яндекс.Диск через
**официальный [REST API Яндекс.Диска](https://yandex.ru/dev/disk/api/)** (обычный `curl`,
без сторонних утилит).

**Режим overwrite-only.** На Диске всегда лежат ровно два файла со стабильными именами,
которые перезаписываются на месте (`overwrite=true`):

| Файл на Диске | Что это |
|---------------|---------|
| `image-rotator-setup-latest.exe` | NSIS-установщик, всегда последняя версия |
| `image-rotator-portable-latest.exe` | портативный exe, всегда последняя версия |

История версий не копится — прямые ссылки на эти файлы постоянны.

Всё бесплатно: GitHub Actions для приватного репозитория даёт 2000 минут/месяц
(один прогон — десятки секунд), Яндекс.Диск — твой.

> ## ⚠️ Важно: почему НЕ rclone
> Яндекс блокирует встроенный OAuth-client_id rclone, и любой запрос отдаёт `401
> Unauthorized`, даже с технически корректным токеном. Поэтому работаем с API напрямую,
> через **собственное OAuth-приложение Яндекса**. Токен, полученный через `rclone config`,
> **не подойдёт** — нужен токен от своего приложения (см. шаг 2).

## Папка и ссылка

- Папка на Диске: **`image-rotator-releases`** (в корне Диска). Если её нет — workflow
  создаст сам.
- Постоянная публичная ссылка на папку: **https://disk.yandex.ru/d/k0EwLxhEl6m78w**
  Эту ссылку даёшь людям — внутри всегда актуальные установщик и портатив.

> **Почему ссылка на папку, а не на файл.** Публичная ссылка на **папку** не меняется
> никогда — внутри файлы можно как угодно обновлять. Плюс мы перезаписываем файлы на
> месте (`overwrite=true`, имена стабильные), так что и прямые ссылки на конкретные
> `*-latest.exe` остаются стабильными.

## Настройка (один раз)

### Шаг 1. OAuth-приложение и токен — уже есть

Токен Яндекс.Диска **переиспользуется** от ранее созданного OAuth-приложения (то же, что
для других проектов). Он привязан к аккаунту, а не к папке, и имеет доступ на запись всего
Диска, поэтому работает и с `image-rotator-releases`. Новое приложение создавать не нужно.

Если когда-нибудь понадобится выпустить токен заново — порядок ниже.

<details>
<summary>Как зарегистрировать OAuth-приложение и получить токен (справка)</summary>

1. [oauth.yandex.ru](https://oauth.yandex.ru/) → **«Создать приложение»**.
2. Название — любое (например, `image-rotator CI`).
3. **Платформы** → **«Веб-сервисы»**, *Redirect URI*:
   `https://oauth.yandex.ru/verification_code`
4. **Доступы (scopes)** → раздел **Яндекс.Диск REST API**:
   - `cloud_api:disk.read` — чтение Диска
   - `cloud_api:disk.write` — запись на Диск
   - `cloud_api:disk.info` — информация о Диске
5. Создать приложение, скопировать **ClientID**.
6. Открыть в браузере (подставив ClientID):
   `https://oauth.yandex.ru/authorize?response_type=token&client_id=ВСТАВЬ_CLIENT_ID`
   Подтвердить доступ → в адресной строке будет `#access_token=y0_...&...`. Скопировать
   значение `access_token` (просто строка, живёт ~1 год).

</details>

### Шаг 2. Положить токен в секреты репозитория

1. GitHub → репозиторий → **Settings** → **Secrets and variables** → **Actions** →
   **New repository secret**:
   - Name: **`YANDEX_DISK_TOKEN`**
   - Secret: строка `access_token`.
2. Сохранить.

Через CLI (токен не попадёт в историю команд):
```powershell
"y0_ВСТАВЬ_ТОКЕН" | gh secret set YANDEX_DISK_TOKEN --repo baslie/image-rotator
```

> Токен хранится **только в настройках репозитория GitHub**, не в коде и не в этом документе.

### Шаг 3. Папка в workflow (если назвал иначе)

В `.github/workflows/release-to-yandex-disk.yml` — переменная `YADISK_DIR`
(по умолчанию `image-rotator-releases`). Путь считается от корня Диска.

## Как выпускать релиз

Сборка — локально (Tauri в облаке не собираем):

```powershell
cd app
npm run tauri:build
# установщик: app/src-tauri/target/release/bundle/nsis/*-setup.exe
# портатив:   app/src-tauri/target/release/image-rotator.exe
```

Создать релиз на GitHub и приложить **оба** файла — это важно: workflow забирает артефакты
из релиза, поэтому портативный exe тоже должен быть приложен.

```powershell
gh release create v1.3.0 `
  (Get-ChildItem app/src-tauri/target/release/bundle/nsis/*-setup.exe).FullName `
  app/src-tauri/target/release/image-rotator.exe `
  --title "v1.3.0" --notes "Что нового в этой версии"
```

Как только релиз опубликован — workflow срабатывает сам. Через ~минуту в папке на
Яндекс.Диске обновятся `image-rotator-setup-latest.exe` и `image-rotator-portable-latest.exe`.

## Ручной прогон (без нового релиза)

GitHub → **Actions** → **«Release → Яндекс.Диск»** → **Run workflow**. Поле `tag` пустое —
возьмётся самый свежий релиз, либо вписать конкретный тег.

CLI:
```powershell
gh workflow run release-to-yandex-disk.yml --repo baslie/image-rotator
```

---

## Проверка и отладка

- Лог прогона: **Actions** → выбрать запуск → шаг «Залить на Яндекс.Диск». В конце
  печатается содержимое папки на Диске.
- **`401` / «Авторизация не прошла»** — токен невалиден/протух или получен через `rclone`.
  Выпусти новый (справка в шаге 1) и обнови секрет. Проверь scopes
  `disk.read` + `disk.write` + `disk.info`.
- **Файл не обновился** — проверь, что к релизу приложен нужный `.exe` (установщик — с
  суффиксом `-setup.exe`, портатив — любой другой `.exe`).
- **Папка не создаётся** — проверь `YADISK_DIR` (путь от корня Диска).
